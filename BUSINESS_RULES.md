# Business Rules — Solo CEO

> **PURPOSE**: This file documents domain-specific rules that code audits and refactors MUST respect.
> Any AI assistant or developer modifying this codebase MUST read this file first.
> Violating these rules causes silent data loss or broken features.

## 1. Task Scopes

Tasks have a `scope` field with three valid values:
- `"work"` — regular kanban tasks (default)
- `"work-memo"` — work memos (shown in WorkMemoList, NOT in kanban)
- `"personal"` — personal memos (shown in personal tab WorkMemoList)

**Rule**: Never hardcode `"work-memo"` when the component receives `scope` as a prop. Use the prop.

## 2. Task Due Dates

The `due` field supports two formats:
- Date only: `"YYYY-MM-DD"` (10 chars)
- Date + time: `"YYYY-MM-DDThh:mm"` (16 chars)

**Rule**: Any `str()` truncation on `due` must allow 16 chars minimum. Any date comparison must use `.slice(0, 10)` to extract only the date portion.

## 3. API PUT Endpoints — Partial Updates

PUT endpoints MUST use partial updates (only write fields present in the request body). Fields not included in the request must NOT be touched.

**Why**: Different UI forms send different subsets of fields. A full overwrite will wipe fields that the form doesn't know about:
- Task edit panel sends `title/client/priority/due/column` but NOT `scope/aiBreakdown/parent_id`
- Client edit form sends billing/contact info but NOT `industry/brand_context` (set during lead conversion)
- Milestone edit form sends `label/amount/percentage` but NOT `invoice_number/payment_method`

**Pattern**:
```js
const patch = {};
if (body.field !== undefined) patch.field = validate(body.field);
// Only update if there's something to update
if (Object.keys(patch).length === 0) return ok({ success: true });
await supabase.from('table').update(patch).eq('id', id);
```

## 4. Subscription Ledger Sync

`syncClientSubscriptionLedger()` regenerates subscription transactions whenever a client is saved.

**Rule**: The sync must NEVER overwrite `status = '已完成'` on existing transactions. If a user has confirmed receipt via the confirm-receipt endpoint, that status must be preserved. Only newly created transactions should get `'待收款 (应收)'`.

## 5. Finance Transaction Sources

The `source` field on `finance_transactions` uses these exact values:
- `"manual"` — user-created transactions (editable/deletable)
- `"subscription"` — auto-generated by subscription ledger sync (NOT `"client_subscription"`)
- `"milestone"` — auto-generated when marking a milestone as paid (NOT `"client_project"`)
- `"project_fee"` — project fee receivable

**Rule**: Source-locked transactions (non-manual) cannot be edited or deleted via the standard PUT/DELETE endpoints. Only dedicated endpoints (confirm-receipt, undo-receipt, mark-paid, undo-paid) can modify them.

## 6. Finance Transaction Statuses

Valid statuses: `"已完成"`, `"待收款 (应收)"`, `"待支付 (应付)"`

**Rule**: These are Chinese strings with special characters (parentheses, spaces). They must match exactly. Validation whitelists must include the full strings.

## 7. Milestone Statuses

- `"pending"` — stored in DB, shown as-is
- `"paid"` — stored in DB, shown as-is
- `"overdue"` — COMPUTED at read time (never written to DB). A pending milestone past its due date.

**Rule**: `"overdue"` must NEVER appear in write-path validation whitelists. It only exists in API GET responses.

## 8. Validation (`enumVal` / `str`)

`enumVal(value, whitelist, fallback)` silently returns `fallback` when value is not in whitelist.

**DANGER**: If the whitelist is incomplete, valid data gets silently replaced with the default. This has caused multiple production bugs.

**Rules**:
- Before adding `enumVal()`, search the ENTIRE codebase for all places that field is set
- Include ALL valid values, not just the ones visible in one component
- For PUT endpoints, prefer partial updates over `enumVal()` with fallback — if a field isn't in the request, don't touch it at all
- `str(value, maxLen)` truncates silently. Check the actual maximum possible length of the data, including auto-generated content

## 9. API Client (`api.ts` vs raw `fetch`)

All internal API calls (`/api/...`) MUST use the `api.ts` utility (`api.get/post/put/del`). Raw `fetch()` is only acceptable for external APIs (OpenAI, Anthropic, DeepSeek, Gemini).

**Why**: `api.ts` goes through `supabase-interceptor.ts` which handles auth, offline fallback, and error normalization. Raw `fetch('/api/...')` also works (interceptor patches global fetch), but `api.ts` provides type safety and consistent error handling.

## 10. Lead Columns

The kanban board has 5 columns: `new`, `contacted`, `proposal`, `won`, `lost`.

**Rule**: There is no `"negotiation"` column. It was removed from the UI. Do not add it to whitelists.

## 11. Payment Methods (Clients)

- `"auto"` — platform auto-debit (Stripe, etc.) — subscription transactions created as `"已完成"`
- `"manual"` — check/cash/transfer — subscription transactions created as `"待收款 (应收)"`, user confirms via confirm-receipt button

## 12. Subscription Timeline

`subscription_timeline` is a JSON array stored as TEXT. Events: `start`, `pause`, `resume`, `cancel`.

**Rule**: Never apply `str()` truncation to this field — it would corrupt the JSON. It can grow unboundedly.
